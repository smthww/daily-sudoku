## 2. 当前功能

- 自动生成数独题目，并在后台检测题目只有一个解。
- 轻松、标准、挑战三档难度。
- 玩家输入昵称后开始游戏，题目出现时才开始计时。
- 支持数字填写、擦除、提示和答案检查。
- 点击数字时突出显示相关行、列、九宫格和相同数字。
- 完成游戏后将昵称、难度、用时、错误次数和提示次数保存到云端。
- 使用提示的成绩会保存，但不参与排行榜。
- 排行榜不按日期和具体题目区分，只按难度统计历史最佳成绩。
- 每位云端匿名用户在每个难度只展示最佳成绩。
- 排行榜在未开始游戏时也可以查看。
- 支持手机浏览器、微信和 iMessage 链接分享。
- 配置了 PWA 图标及网页离线缓存。

当前 `scores` 安全规则：

```json
{
  "read": true,
  "create": "auth != null && (doc._openid == auth.uid || doc._openid == auth.openid)",
  "update": false,
  "delete": false
}
```

规则含义：

- 所有人可以读取排行榜。
- 已匿名登录的网页或小程序用户可以创建自己的成绩。
- 普通用户不能修改或删除已有成绩。
- 管理员仍可通过 CloudBase 控制台管理数据。

## 6. 成绩数据和排名规则

每条成绩主要包含：

```text
nickname          玩家昵称
difficulty        easy / medium / hard
elapsedSeconds    完成秒数
mistakes          检查答案时的错误次数
hints             使用提示次数
ranked            是否参与排名
userId            CloudBase 匿名用户编号
dateKey           完成日期，保留用于未来功能
puzzleId          题目标识，保留用于问题追踪
createdAt         云端提交时间
```

排行榜处理规则：

1. 读取历史成绩，不限制日期和具体题目。
2. 按轻松、标准、挑战三个难度分别展示。
3. 排除 `ranked=false` 的提示成绩。
4. 同一用户只保留该难度的最好成绩。
5. 优先按用时从短到长排序，用时相同时再比较错误次数。

## 7. 已解决的重要问题

### 成绩显示已保存但数据库为空

原因：数据库规则只判断了小程序使用的 `auth.openid`，网页匿名用户使用 `auth.uid`，成绩实际被拒绝。

处理：规则同时兼容 `auth.uid` 和 `auth.openid`；`cloud.js` 也会检查写入结果中的记录编号，不再误报成功。

### 更新后页面混用旧脚本

原因：Service Worker 缓存了旧版 CSS 和 JavaScript。

处理：核心文件使用 `?v=版本号`，并同步提高 `CACHE_NAME`。以后每次修改核心代码都要更新两处版本号。

### 重新打开后看不到成绩

原因：最初排行榜绑定当天、难度和具体题目，并且只在开始游戏后加载。

处理：改成页面打开即加载的历史总排行榜，只按难度划分。

## 8. 发布流程

### 推荐流程

1. 在本地修改代码。
2. 运行语法和格式检查。
3. 外观修改先提供预览，确认后再上线。
4. 功能修改先在测试页面验证。
5. 将生产文件部署到 CloudBase。
6. 在正式域名验证真实行为和数据库结果。
7. 创建 Git 提交并推送到 GitHub。

### CloudBase 部署

CloudBase CLI 已通过 Homebrew 安装。部署时应上传生产文件，不要上传 `.git` 文件夹。

示例：

```bash
tcb hosting deploy <生产文件目录> -e daily-sudoku-d3g3cccua4adfaf16
```

### GitHub 版本保存

```bash
git status
git add <本次确认修改的文件>
git commit -m "简短说明本次修改"
git push origin main
```

不要使用不明确的批量删除、强制重置或覆盖历史命令。

## 10. 安全和维护提醒

- 当前网站是公开链接，任何获得网址的人都能使用，链接也可以继续转发。
- 排行榜公开显示昵称和成绩，玩家不应使用手机号等敏感信息作为昵称。
- GitHub 仓库目前公开，代码可以被查看和复制，但仓库中没有密码或密钥。
- 当前防作弊适合朋友间娱乐，不适合奖金比赛；懂技术的人仍可能伪造客户端请求。
- 匿名用户换设备或清除浏览器数据后，可能被识别为新用户。
- 需要关注 CloudBase 体验套餐的到期时间、数据库用量和请求额度。
- 如果以后公开推广，建议增加服务端成绩校验、提交频率限制和后台管理功能。
